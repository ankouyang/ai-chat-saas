# 消息缓存时序图

本文档基于当前项目已经接入 Upstash Redis 缓存后的实现，重点说明三件事：

- 聊天页历史消息是如何优先走缓存读取的
- 模型上下文最近 12 条消息是如何单独缓存的
- 新消息写库后，缓存是如何同步更新的

## 关键文件

- `lib/message-cache.ts`
- `lib/redis.ts`
- `app/chat/page.tsx`
- `lib/ai.ts`
- `app/api/chat/route.ts`
- `lib/db.ts`

## 一、聊天页历史消息：全量缓存读取链路

```text
用户
  |
  | 1. 打开 /chat?chatId=xxx
  v
app/chat/page.tsx
  |
  | 2. 先确认 activeChat 是否属于当前用户
  | 3. 调用 listChatMessages(chatId)
  v
lib/message-cache.ts -> listChatMessages(chatId)
  |
  | 4. 生成全量缓存 key
  |    chat:messages:full:${chatId}
  | 5. readMessagesFromRedis(cacheKey)
  v
Upstash Redis
  |
  | 6. 如果 key 命中，直接返回完整消息数组
  v
lib/message-cache.ts
  |
  | 7. 命中缓存：
  |    直接返回 cached messages
  |
  | 8. 未命中缓存：
  |    queryMessagesFromDatabase(chatId)
  v
Prisma / Database
  |
  | 9. 查询当前 chatId 的全部消息
  |    orderBy createdAt asc
  v
lib/message-cache.ts
  |
  | 10. 把数据库结果写回 Redis
  |     EX = MESSAGE_CACHE_TTL_MS
  | 11. 返回消息数组给 page.tsx
  v
app/chat/page.tsx
  |
  | 12. 映射成 initialMessages
  | 13. 交给 ChatWorkspace 首屏渲染
  v
用户看到当前会话的历史消息
```

## 二、模型上下文：最近 12 条消息缓存链路

```text
app/api/chat/route.ts
  |
  | 1. 用户消息先写入 Message 表
  | 2. 调用 generateChatReplyStream(chatId, onChunk)
  v
lib/ai.ts
  |
  | 3. 调用 listRecentChatMessages(chatId, 12)
  v
lib/message-cache.ts -> listRecentChatMessages(chatId, 12)
  |
  | 4. 生成最近消息缓存 key
  |    chat:messages:recent:12:${chatId}
  | 5. readMessagesFromRedis(cacheKey)
  v
Upstash Redis
  |
  | 6. 如果 key 命中，直接返回最近 12 条消息
  v
lib/message-cache.ts
  |
  | 7. 命中缓存：
  |    直接返回 recent messages
  |
  | 8. 未命中缓存：
  |    queryMessagesFromDatabase(chatId, 12)
  v
Prisma / Database
  |
  | 9. 查询最近 12 条消息
  |    orderBy createdAt desc
  |    take 12
  v
lib/message-cache.ts
  |
  | 10. 反转顺序，恢复成时间正序
  | 11. 写回 recent cache key
  | 12. 返回给 lib/ai.ts
  v
lib/ai.ts
  |
  | 13. 把最近 12 条历史消息
  |     + system prompt
  |     组装成模型 messages
  v
模型请求继续向下执行
```

## 三、新消息写入后：缓存同步更新链路

```text
用户
  |
  | 1. 在聊天页发送一条消息
  v
app/api/chat/route.ts
  |
  | 2. prisma.message.create(...)
  |    写入 user message
  | 3. appendChatMessageToCache(userMessage)
  v
lib/message-cache.ts -> appendChatMessageToCache(message)
  |
  | 4. 同时读取两个缓存 key：
  |    - chat:messages:full:${chatId}
  |    - chat:messages:recent:12:${chatId}
  v
Upstash Redis
  |
  | 5. 返回已存在的缓存值
  v
lib/message-cache.ts
  |
  | 6. 如果 full cache 存在：
  |    追加新消息并重新 set
  | 7. 如果 recent cache 存在：
  |    追加新消息后只保留最后 12 条
  | 8. 两个 key 都重新设置 TTL
  |
  | 9. 如果某个 key 当前不存在：
  |    不主动创建，等下次读取时再从数据库回填
  v
app/api/chat/route.ts
  |
  | 10. 模型流式返回结束后
  |     assistant message 写库
  | 11. appendChatMessageToCache(assistantMessage)
  v
缓存与数据库尽量保持同步
```

## 四、缓存 key 设计

当前项目用了两类缓存 key：

- 全量历史消息：
  `chat:messages:full:${chatId}`
- 最近 12 条模型上下文：
  `chat:messages:recent:12:${chatId}`

这样拆的原因是：

- 聊天页首屏需要完整历史消息
- 模型请求只需要最近窗口，不必读取整段会话
- 两条链路命中模式不同，拆 key 更容易控制 TTL 和写入策略

## 五、TTL 策略

缓存 TTL 在 `lib/message-cache.ts` 中统一处理：

- 默认 `15000ms`
- 可通过 `MESSAGE_CACHE_TTL_MS` 覆盖
- 写入 Redis 时统一换算成秒后传给 `EX`

效果是：

- 热会话在短时间内会复用缓存
- 长时间不访问的会话缓存会自然过期
- 过期后再次读取会自动回源数据库并重新填充缓存

## 六、为什么现在不是“完全不查库”

当前实现是“缓存优先，数据库兜底”，不是“完全只读缓存”。

也就是说：

- 首次访问某个会话时，通常还是会查一次数据库
- 缓存过期后，也会再次查数据库
- 只有缓存命中时，才会直接跳过数据库

这样做的原因是：

- Redis 只做性能优化，不做唯一数据源
- 数据库仍然是最终真实来源
- 即使缓存丢失或失效，聊天数据也不会丢

## 七、一句话总结

当前项目的消息缓存链路是：

`聊天页读取历史消息 -> 先查 full cache，未命中再查库并回填`

`模型读取上下文 -> 先查 recent:12 cache，未命中再查库并回填`

`新消息写库 -> 尝试同步更新 full cache 和 recent:12 cache -> 下次读取优先命中 Redis`
