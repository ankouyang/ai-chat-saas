# 大模型接入与流式聊天时序图

本文档基于当前项目已经接入真实大模型后的实现，重点说明两件事：

- `lib/ai.ts` 是如何把数据库里的历史消息组装成模型上下文，并发给大模型
- 聊天页是如何从“提交后整页刷新”改成“按 chunk 实时追加回复”的

## 关键文件

- `lib/ai.ts`
- `lib/chat.ts`
- `app/api/chat/route.ts`
- `components/chat-workspace.tsx`
- `app/chat/page.tsx`
- `lib/auth.ts`
- `lib/db.ts`

## 一、模型接入：服务端如何组织上下文并请求大模型

```text
app/api/chat/route.ts
  |
  | 1. 接口确认当前用户已登录
  | 2. 解析请求体里的 content / chatId
  | 3. ensureChatForUser() 确保当前会话存在，且属于当前用户
  | 4. 先把用户消息写入 Message 表
  v
lib/ai.ts -> generateChatReplyStream(chatId, onChunk)
  |
  | 5. 根据 chatId 查询最近 12 条消息
  |    prisma.message.findMany({
  |      where: { chatId },
  |      orderBy: { createdAt: "asc" },
  |      take: 12
  |    })
  |
  | 6. 把数据库消息映射成模型需要的 messages 数组
  |    - 第一条固定是 system prompt
  |    - 后面依次拼接 user / assistant 历史消息
  |
  | 7. resolveBaseUrl() 解析模型基础地址
  | 8. resolveModelCandidates() 解析主模型 + 回退模型
  | 9. requestChatCompletion() 向
  |    ${OPENAI_BASE_URL}/chat/completions
  |    发起 POST 请求
  |    body:
  |    {
  |      model,
  |      temperature: 0.7,
  |      messages,
  |      stream: true
  |    }
  v
大模型 / 代理服务
  |
  | 10. 按流式返回 delta 内容
  v
lib/ai.ts -> readStreamingChatCompletion(response, onChunk)
  |
  | 11. 逐块读取 response.body
  | 12. 解析每一行 SSE data:
  | 13. 取出 chunk.choices[0].delta.content
  | 14. 一边累积 assistantContent
  | 15. 一边把 delta 通过 onChunk 回调抛给上层
  v
app/api/chat/route.ts
  |
  | 16. 最终拿到完整 assistantContent
  | 17. 把 assistant 消息写入 Message 表
  | 18. 更新 Chat.title
  | 19. revalidatePath("/chat")
```

## 二、流式聊天：前端如何边收边显示

```text
用户
  |
  | 1. 在聊天输入框输入内容，点击“发送”
  v
components/chat-workspace.tsx
  |
  | 2. handleSubmit(event)
  | 3. event.preventDefault()，阻止传统表单跳转
  | 4. 本地立刻插入：
  |    - 一条 user 消息
  |    - 一条空 assistant 占位消息
  | 5. fetch("/api/chat", {
  |      method: "POST",
  |      body: { content, chatId }
  |    })
  v
app/api/chat/route.ts
  |
  | 6. 返回 text/event-stream
  | 7. 先发 meta 事件：
  |    event: meta
  |    data: { chatId }
  | 8. 再持续发 delta 事件：
  |    event: delta
  |    data: { content: "..." }
  | 9. 结束时发 done 事件
  | 10. 失败时发 error 事件
  v
components/chat-workspace.tsx
  |
  | 11. response.body.getReader() 读取流
  | 12. TextDecoder + buffer 按 SSE 事件边界拆包
  | 13. 收到 meta:
  |     - setChatId(chatId)
  |     - router.replace(`/chat?chatId=${chatId}`)
  | 14. 流式过程中跳过一次由新 props 引发的消息重置
  |     - 避免本地正在增长的 assistant 内容被 initialMessages 覆盖
  | 15. 收到 delta:
  |     - 立刻追加到当前 assistant 占位消息 content
  |     - 用户能在页面上实时看到文本增长
  | 16. 收到 done:
  |     - 用最终完整文本覆盖一次 assistant 内容
  | 17. 收到 error:
  |     - 把错误文案显示到 assistant 消息气泡和输入框下方
  | 18. 流结束后 router.refresh()
  v
app/chat/page.tsx
  |
  | 19. 服务端重新查库
  | 20. 侧边栏 chats / 当前 activeChat 与数据库最终状态同步
  v
用户看到“实时增长的回复” + 最终一致的数据
```

## 三、为什么模型接入要这样拆层

### 1. `lib/ai.ts` 只负责模型语义，不负责页面交互

`lib/ai.ts` 的职责是：

- 读取会话历史
- 组装模型 messages
- 选择 base URL 和 model
- 处理回退模型
- 解析流式或非流式响应

这样做的原因是：模型接入逻辑不应该和页面组件、路由响应混在一起。否则之后你想换模型供应商、换响应格式、加日志、加模型回退，都会改到聊天页面本身。

### 2. `app/api/chat/route.ts` 负责“会话写库 + 流式转发”

这个文件的职责是：

- 校验用户登录
- 校验/创建 chat
- 写入用户消息
- 调用 `generateChatReplyStream`
- 把模型 chunk 转成前端能消费的 SSE 事件
- 最后把 assistant 消息落库

这样拆的原因是：路由层最适合处理“HTTP 请求/响应”和“数据库副作用”，而不适合承担模型协议细节。

### 3. `components/chat-workspace.tsx` 负责即时交互

这个组件的职责是：

- 先做本地 optimistic update
- 接收接口流
- 边收边改本地消息状态
- 在完成后再轻量同步服务端状态

这样做的原因是：真正的聊天体验必须由客户端状态驱动。只靠服务端 `redirect + SSR`，永远只能“最终显示”，做不出过程感。

## 四、和旧实现相比，核心变化是什么

旧实现：

```text
form action={sendMessageAction}
  -> 服务端写库
  -> redirect("/chat?chatId=xxx")
  -> page.tsx 重新查库
  -> 页面一次性显示最终消息
```

新实现：

```text
chat-workspace.tsx fetch("/api/chat")
  -> route.ts 先写 user message
  -> ai.ts 流式调用模型
  -> route.ts 把 delta 逐块推给前端
  -> chat-workspace.tsx 逐块更新 assistant 气泡
  -> 结束后 route.ts 落库 + page.tsx 刷新同步
```

本质区别：

- 旧实现是“服务端完成全部工作后，页面再统一看到结果”
- 新实现是“服务端边拿模型输出，边把结果推给前端，前端边显示”

## 五、代码阅读顺序建议

如果你要按逻辑读这套代码，建议顺序是：

1. `components/chat-workspace.tsx`
先看前端为什么不再用 form action，而是改用 `fetch + reader`

2. `app/api/chat/route.ts`
再看接口如何把模型输出包装成 SSE 事件

3. `lib/ai.ts`
最后看模型请求本身：上下文怎么取、模型怎么选、流怎么解析

4. `lib/chat.ts`
补充理解 chatId 校验和自动建会话逻辑

## 六、一句话总结

当前这套“大模型接入 + 流式聊天”的核心链路是：

`前端提交消息 -> 服务端校验/建 Chat -> 用户消息先入库 -> lib/ai.ts 带历史上下文请求模型 -> 服务端把 delta 逐块推给前端 -> 前端按 chunk 追加 assistant 气泡 -> 结束后 assistant 消息落库并 refresh 做最终一致性同步`
